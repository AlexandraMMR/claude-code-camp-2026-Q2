# Agent Architecture Exploration: Coding Agent as MUD Player

## Goal

Our goal was to determine whether a simple coding agent can cost-effectively play a MUD (tbaMUD on `localhost:4000`) by finding the bakery and listing its menu. I used Antigravity as my IDE and am testing various Claude models.

## Observations

- **Temporary connection scripts**: The agent created multiple temporary Python and PowerShell scripts (`mud_client.py`, `mud_connect.ps1`) on each attempt to manage the raw TCP/socket connection, rather than using any reusable abstraction.
- **Unrelated file reads**: The agent read `CLAUDE.md`, `data/player.md`, and `data/world.md` at the start — the memory files were empty and added no value, consuming unnecessary tokens.
- **Login flow failures**: The agent initially confused the login sequence: it sent `Y` (as a name confirmation step) which was consumed as the password, causing a failed login. It took two full connection attempts before the correct flow (`username → password → RETURN → 1`) was discovered from trial-and-error output.
- **Poor feedback loop**: Because each script run was a batch execution (connect, send commands, disconnect), failed steps produced no intermediate output. The agent had limited visibility into exactly which command caused the login failure until the full output was parsed.
- **Model intelligence**: Switching to a more capable model (Claude Sonnet with extended thinking) did not resolve the core issue — the problem was architectural (no reliable login abstraction), not a reasoning deficit.
- **Navigation progress**: The agent did successfully use `where baker` to locate the baker NPC and identify the room name (`The Bakery`), but navigating there via Main Street is still in progress.

## Technical Conclusion

A coding harness or default agent with a coding focus is not a good fit for operating the MUD. The deterministic login and connection flow should be handled by a reusable MUD SDK or MCP server rather than regenerated by the model on every attempt.

- **State storage**: Markdown files (`player.md`, `world.md`) are too unstructured for complex, evolving player and world state; a structured format (JSON, SQLite) would be more reliable, it still didn't update any of the files or memory after the actions though.
- **Goal status**: The bakery was located (`where baker` → `The Bakery`), but the menu was not retrieved before this summary was written. Navigation to the room and a `list` command remain incomplete, it seemed to get confused on which direction to go, didn't have memory for the mapping, it would go one direction and then back the same direction. It also had an issue staying connected, it would timeout or lose connection and need to reconnect, which reset its progress or have to re-write the scripts and run them again. It could connect and login though eventually, so that was a step forward.
